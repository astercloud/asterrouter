# AsterRouter V3 路线图

## 版本定位

V3 是“稳定 API 企业网关”版本。它在 V2 已有 Workspace Key、Gateway Model、显式 Route、调度 Trace、用量和插件基础上，优先补齐企业身份与治理控制面。

V3 不扩展上游账号接入边界。企业可以采购第三方网关或自行部署兼容服务处理上游账号与渠道；AsterRouter 只通过公开、稳定的标准 API 与这些系统协作。

## 目标能力

- 飞书 / Lark、钉钉企业登录、区域隔离、声明校验和自动入职；保留 OIDC 企业 SSO 兼容入口。
- GitHub / Google 作为可选辅助登录源；LinuxDO、微信和支付不进入 V3 范围。
- 用户、部门、Group 与资源级 RBAC。
- Workspace Key 生命周期、模型策略、预算和配额。
- 用量、成本、审计、数据保留与导出。
- 员工 Portal 的个人 Key、模型、额度和用量视图。
- 默认 Developer 可创建、轮换和禁用本人个人 Key；Key、用量、Trace 与告警按 `OwnerUserID` 隔离，跨用户资源统一按不存在处理。
- 单实例私有化部署的健康检查、诊断、备份恢复和升级校验。

## 阶段划分

### V3.1 身份接入

- OIDC Discovery、JWKS、Authorization Code + PKCE。
- `issuer + subject` 外部身份绑定。
- 多身份资料、安全绑定与解绑；绑定交易服务端关联当前用户并复用 OAuth state、PKCE/nonce，禁止移除账户最后一个可用登录方式。
- 首次登录自动创建用户、映射部门和默认角色。
- 禁用用户拒绝登录，管理员授权覆盖默认映射。
- 持久化会话版本撤销：登出、改密、密码重置、禁用和角色变化后，已签发令牌立即失效。
- 认证 Provider 配置保存后返回明确的运行态重启要求与变更原因；当前进程不伪装成已热更新，管理员可在维护窗口重启生效。

### V3.2 企业治理

- 部门、Group、角色和资源范围。
- 组织 Group 使用独立 CRUD、成员唯一归属约束与审计；因其可跨部门，仅全局身份管理员可管理，部门管理员不可借此越权关联其他部门用户。
- 角色绑定支持部门范围；部门管理员的用户与个人 Key 列表、策略查看和写操作按部门过滤，共享 Key 与其他部门对象不可见且不可修改。
- 部门范围继续传递到用量、成本分摊、Trace、告警、汇总及同步/异步 CSV 导出，并与显式查询条件取交集；异步任务记录创建者并限制列表、详情和下载所有权，全局审计仍不向部门管理员开放。
- Key 创建、轮换、禁用、过期、模型白名单、QPS 和预算策略。
- 配额超额阻断、告警去重和策略决策审计。
- 有效 Token 配额与月度预算在 80% 生成去重预警、达到上限升级为严重告警，并由网关前置策略按 `overage_action` 阻断或放行；拒绝请求写用量与策略快照 Trace。

### V3.3 数据与运维

- 用量与成本按用户、部门、组织 Group、Key 和模型聚合；组织 Group 使用独立成员表，每个用户最多归属一个 Group，避免成本重复归集，不与路由 Group 或 Operator 使用方分组混用。
- 审计检索、导出、脱敏和保留期清理。
- `data_retention_days` 在服务启动时真实执行；管理员也可二次确认后手动清理并查看分类计数。清理覆盖用量、Trace、已解决告警和旧审计，活动告警与本次清理审计证据保留。
- 健康检查、诊断包、PostgreSQL 备份恢复演练和升级前校验。
- 可取消的固定间隔自动备份：动态读取 1-720 小时间隔，复用手动备份、S3 上传及本地/远端保留清理链路，成功与失败均写系统审计；不引入 Cron 解析依赖。

## 非目标

- 不实现上游账号 OAuth、Token 刷新、账号池、采集、余额抓取或自动续期。
- 不实现网页态、Cookie、浏览器自动化、逆向 API 或私有管理 API。
- 不新增 Provider 私有连接器；正式接入以 OpenAI Compatible、Anthropic Messages、Gemini API 为主。
- 不实现充值、支付、订阅、余额售卖或公开中转运营闭环。
- 兼容数据模型中的历史月费字段不可新增非零值；人工流水仅允许企业额度分配、收回与成本修正。
- 不新增 Batch、异步上游任务、Realtime 或其他需要额外上游状态模型的协议。
- 不以多实例 HA 或 Kubernetes 作为 V3 交付目标。

## 验收标准

- 企业可通过 OIDC 登录，首次登录可按声明自动完成用户与部门入职。
- 飞书中国版、Lark 国际版和钉钉按各自官方企业身份链登录，不混用区域端点。
- 用户登出、改密、被禁用或角色调整后，旧 Bearer Token 不再具有访问权限。
- 部门和角色只能访问授权的用户、Key、策略、用量和审计范围。
- Key 策略、预算和配额在网关请求中真实生效，并产生可追踪审计证据。
- 员工 Portal 不泄露其他用户数据、Provider Secret 或外部网关内部状态。
- 资源级角色绑定的访问上下文必须明确区分全局授权与单资源域授权，不得将 Developer 的资源绑定标记为全局权限。
- 使用记录和审计日志可导出、可脱敏，并按保留策略清理。
- 外部中转站仅通过标准 API 接入；其内部账号和运营能力变化不影响 AsterRouter 的产品契约。

## 参考项目的吸收边界

- 吸收成熟网关产品的标准协议覆盖、错误分类、公开 API 兼容测试和多机部署安全经验。
- 吸收成熟管理后台的信息组织、状态展示、脱敏和可观测性经验。
- 不吸收两者的账号采集、上游 OAuth 自动化、内部管理 API、支付售卖和历史兼容包袱。
